# Codex Deck ユースケース一覧

## 1. 文書情報

| 項目 | 内容 |
| --- | --- |
| 対象 | Codex Deck MVPの利用シナリオと例外処理 |
| 前提 | [要件定義書](CODEX_DECK_REQUIREMENTS.md)および[公式機能調査](../research/CODEX_OFFICIAL_CAPABILITY_RESEARCH.md) |
| 役割 | 利用者操作、Deckの責務、Codexの正本、PoC境界を具体化する。 |

## 2. 共通前提

- 利用者はTailscale経由でCodex Deckに到達し、Deckの認証/端末確認を完了している。
- App Serverはブラウザから直接到達できず、Deck BridgeだけがWindows PC内の標準入出力で接続する。
- 「セッション」はCodex Threadを指す。DeckはThread本文やapproval policyの正本を複製しない。
- ファイル/コード表示は読み取り専用である。利用者はCodexへの引用と依頼はできるが、ブラウザ上で直接編集できない。
- CLI/VS CodeとDeckの相互Thread操作は、PoC-2に合格するまで条件付き機能として表示する。

## 3. ユースケース

### UC-01 スマホから新規作業を依頼する

| 項目 | 内容 |
| --- | --- |
| 主アクター | 利用者（スマートフォン） |
| 目的 | 許可済みworkspaceでCodexの新規ThreadとTurnを開始する。 |
| 関連要件 | FR-WS-01〜06、FR-SS-01〜06、FR-CX-01〜05 |
| 事前条件 | workspaceが登録済み、または許可ルート内から明示追加済み。対象workspaceにActive workがない。 |
| 成功条件 | Codex ThreadとTurnが開始され、DeckがThread ID、workspace、アクセスモード、最初のItemを表示する。 |

**基本フロー**

1. 利用者はホームからworkspaceを選ぶ。
2. DeckはGit状態、既存セッション、Active workの有無を表示する。
3. 利用者は「新規作業」を選び、公式に利用可能なアクセスモードを確認する。
4. 利用者は全画面composerまたは通常composerに複数行の依頼を入力し、必要ならファイル/行/選択範囲を引用する。
5. DeckはIME変換中でないこと、オンラインであること、入力プレビューを確認できることを保証する。
6. 利用者が送信する。
7. Deck Bridgeは公式`thread/start`と`turn/start`を実行し、返却Thread/TurnとストリーミングItemを表示する。
8. Deckはworkspace lockをActive workに設定し、ブラウザを閉じてもBridgeが処理を継続する。

**代替・例外フロー**

- 対象workspaceに実行中Turnがある場合、Deckは新規実行を開始せず、実行中セッションへ移動、停止、または終了待ちを案内する。
- ネットワーク切断中は送信ボタンを無効化し、「未送信下書き」と明示する。送信済みと見せない。
- Thread作成後にTurn開始が失敗した場合、Threadの扱いを確認可能な状態でエラー表示し、無条件に再送しない。

### UC-02 タブレットから実行中作業を監視する

| 項目 | 内容 |
| --- | --- |
| 主アクター | 利用者（タブレット縦/横） |
| 目的 | 実行中Threadの発言、計画、コマンド、ファイル変更、テストを同時に確認する。 |
| 関連要件 | FR-RT-01〜06、FR-FV-01〜04、FR-EX-01〜05、UI要件 |
| 事前条件 | 対象ThreadがDeck Bridgeにより監視中、または再読込可能である。 |
| 成功条件 | 公式Itemに基づく進捗を、順序を保って表示できる。 |

**基本フロー**

1. 利用者は通知またはセッション一覧から実行中Threadを開く。
2. 縦向きでは左ペインを「会話/作業」切替、右ペインをコード/diff/テストにする。
3. 横向きでは左にファイル、中央にコード/diff、右に会話/作業を表示する。
4. Deckはagent message、tool progress、command、file change、test関連Itemを受信順に表示する。
5. 利用者は任意のファイル、行、diff、コマンドを選択し、関連する会話Itemへジャンプする。
6. 利用者が下部ターミナル/テストペインを開くと、長い出力は仮想スクロールで表示する。

**代替・例外フロー**

- 表示端末がbackground化・回転した場合、レイアウトと最終event IDを保持し、復帰時に再同期する。
- 未知Itemは削除せず「未対応イベント」と表示し、技術情報は折畳む。
- 一時切断では「再接続中」と表示し、Codex作業が継続中であることとブラウザ接続が別であることを区別する。

### UC-03 Codexの質問へ回答する

| 項目 | 内容 |
| --- | --- |
| 主アクター | 利用者 |
| 目的 | Codexが明示した質問または構造化入力要求へ、誤送信なく回答する。 |
| 関連要件 | FR-CX-01、FR-CX-06、FR-RT-01、FR-NF-01 |
| 事前条件 | Thread/Turnに紐付く質問または`tool/requestUserInput`相当の要求をDeckが受信している。 |
| 成功条件 | 回答は正しい要求IDに一度だけ送られ、解決/完了イベントにより状態が更新される。 |

**基本フロー**

1. Deckは質問待ちを画面ヘッダ、セッション一覧、通知で強調する。
2. 利用者は質問内容、選択肢、Thread/Turn、期限（ある場合）を確認する。
3. 利用者は回答を入力/選択し、送信する。
4. Deckは回答を対象の公式server requestへ関連付けて送信する。
5. `serverRequest/resolved`または後続Itemにより質問待ち状態を解除する。

**例外**

- 回答前にTurnが完了・停止・要求解消された場合、Deckは送信を止め、要求が解消済みであることを表示する。
- 同一要求を別クライアントが解決した可能性がある場合、二重送信せず最新状態を取得する。

### UC-04 承認要求へ応答する

| 項目 | 内容 |
| --- | --- |
| 主アクター | 利用者 |
| 目的 | Codex公式のコマンド、ファイル変更、ネットワーク等の承認要求を確認し、公式decisionを返す。 |
| 関連要件 | FR-AP-01〜05、FR-NF-01、SEC-07 |
| 事前条件 | App Serverから`item/*/requestApproval`等を受信している。 |
| 成功条件 | 承認/拒否は対象Thread/Turn/Itemに紐付き、公式decisionで処理され、監査記録に残る。 |

**基本フロー**

1. Deckは承認待ちを優先表示し、必要ならPWA/ntfy通知を送る。
2. 利用者は理由、コマンド、cwd、変更ファイル、ネットワーク宛先、アクセスモードを確認する。
3. Deckは公式が提示した選択肢のみを表示する。
4. 利用者は`accept`、`acceptForSession`、`decline`、`cancel`等を明示選択する。
5. Deckはrequest IDを含む応答をApp Serverへ送信し、決定を補助監査ログに記録する。
6. `serverRequest/resolved`と`item/completed`を受け、結果を会話と作業ログに反映する。

**例外**

- network approvalでは通常のshellコマンド承認と誤解させず、host/protocol/portを主表示する。
- 実行中に別クライアントが決定した場合は、失敗を再送せず最新状態に更新する。
- フルアクセス中は状態バッジを隠さない。Deckは独自の強制制限を追加しない。 |

### UC-05 ファイルを閲覧し、行を引用する

| 項目 | 内容 |
| --- | --- |
| 主アクター | 利用者 |
| 目的 | workspace内のコードを読み取り専用で確認し、ファイル/行/選択範囲をCodexへの依頼に添える。 |
| 関連要件 | FR-FV-01〜04、SEC-05 |
| 事前条件 | workspaceが許可済み。ファイルは閲覧deny/mask規則に抵触しない。 |
| 成功条件 | ファイル内容と行番号を安全に表示し、引用がcomposerへ構造化挿入される。 |

**基本フロー**

1. 利用者はファイルdrawer/ツリーから対象を選択する。
2. Deckは`.gitignore`、除外規則、巨大/バイナリ/文字コード判定を適用する。
3. コードビューに行番号、ハイライト、折返し、検索、フォント調整を表示する。
4. 利用者は行または範囲を選び、「Codexに引用」を選択する。
5. Deckはworkspace相対パス、行範囲、選択本文をcomposerへ追加し、利用者が送信前に編集・確認できるようにする。

**例外**

- `.env`、credential、SSH key等は内容を返さず、制限理由のみを表示する。
- symlinkが許可範囲外を指す場合は追跡しない。
- 巨大ファイルはchunk表示またはダウンロード禁止/プレビュー不可を明示する。

### UC-06 Git差分を確認し、修正依頼する

| 項目 | 内容 |
| --- | --- |
| 主アクター | 利用者 |
| 目的 | Codex作業による差分を端末に合った表現で読み、選択箇所を修正依頼へ引用する。 |
| 関連要件 | FR-GT-01〜04、FR-CX-01 |
| 事前条件 | workspaceがGitリポジトリ。Git状態が取得可能。 |
| 成功条件 | diffは誤った編集可能UIを持たず、引用は正しいファイル/hunk/行にリンクされる。 |

**基本フロー**

1. 利用者は「差分」タブを開き、未コミット/staged/unstaged等の状態を確認する。
2. スマホではインラインdiff、タブレット/PCではside-by-sideを既定表示する。
3. 利用者はファイル・hunk・行を移動し、必要な前後contextを開く。
4. 利用者は「この差分を引用」を選択する。
5. Deckはファイル、行、hunkの参照をcomposerへ挿入する。
6. 利用者は「この変更でエラー処理を追加して」等の依頼を送る。
7. 関連Threadの会話Itemと差分を相互リンクする。

**例外**

- binary/rename/削除では、表示可能なメタ情報だけを示し、文字列diffを捏造しない。
- 大きなdiffはchunkごとに読み、未読件数を表示する。
- 人間によるhunk編集、stage/unstage、commitの直接操作はMVP対象外。 |

### UC-07 テスト失敗を確認する

| 項目 | 内容 |
| --- | --- |
| 主アクター | 利用者 |
| 目的 | Codexが実行したテストの失敗を、要約と原ログの双方から確認する。 |
| 関連要件 | FR-EX-01〜05、FR-RT-01 |
| 事前条件 | Codexがテストまたは一般コマンドを実行し、Item/出力をDeckが受信している。 |
| 成功条件 | 失敗情報、終了コード、時刻、関連ファイル、原ログがアクセス可能である。 |

**基本フロー**

1. Deckは終了コードまたはparser結果からテスト失敗を検出する。
2. 利用者に、成功/失敗/skip/時間、失敗一覧、最初のerrorを要約表示する。
3. 利用者はstack traceを展開し、ファイル/行リンクでコードまたは差分へ移動する。
4. 利用者は「Codexの説明を開く」または失敗箇所を引用して次の修正依頼を作る。
5. 必要なら「全ログ」を仮想化表示し、コピー/検索する。

**例外**

- 出力形式が認識できない場合は「一般コマンド結果」として終了コードと生ログを正しく表示する。成功/失敗数を推測しない。
- 再実行はDeckが直接コマンドを起動せず、Codexに明示依頼する。 |

### UC-08 バックグラウンド実行の完了通知を受ける

| 項目 | 内容 |
| --- | --- |
| 主アクター | 利用者 |
| 目的 | ブラウザを閉じた後も継続したCodex作業の重要結果へ速やかに戻る。 |
| 関連要件 | FR-NF-01〜03、FR-RC-01〜02 |
| 事前条件 | 作業開始後、Deck Bridgeと通知adapterが稼働している。 |
| 成功条件 | 通知は重複を抑止し、対象Threadの最新状態へ遷移する。 |

**基本フロー**

1. 利用者は作業を開始し、ブラウザ/PWAを閉じる。
2. Deck BridgeはCodex Item/Turnを継続受信し、Browser接続とは無関係に保存する。
3. `turn/completed`、質問、承認、エラー、テスト失敗等の重要イベントで通知判定する。
4. 設定に従いアプリ内/PWA/ntfyへ一度だけ配送する。
5. 利用者が通知を開くと、Deckは認証後に対象Threadへ遷移し、欠落イベントを同期する。

**例外**

- PWA pushが未許可・OS制約で利用不可なら、アプリ内通知履歴とntfy（設定時）を代替とする。
- 通知配送失敗は作業失敗と同一視せず、通知ログへ記録し次回アプリ起動時に未読表示する。 |

### UC-09 CLIで開始したセッションをWebで確認・再開する

| 項目 | 内容 |
| --- | --- |
| 主アクター | 利用者（CLI利用後、Deck利用） |
| 目的 | CLI起点のThreadがApp Serverで検出可能な場合に、Deckから読み/再開する。 |
| 関連要件 | FR-SS-02〜04、PoC-2 |
| 事前条件 | PoC-2でCLI→App Server可視性が合格し、同じCodex環境/設定範囲である。 |
| 成功条件 | 既存Threadとworkspace紐付けを誤認なく表示し、公式`thread/resume`で続行できる。 |

**基本フロー**

1. 利用者はDeckでworkspaceを開く。
2. Deckは公式`thread/list`/`thread/read`の結果を検索・filter表示する。
3. 利用者はCLI起点と分かる既存Threadを選択する。
4. Deckは履歴・状態を読み、利用者が「再開」を選択した場合に公式resumeを行う。

**例外**

- Threadが一覧に現れない、workspace/cwdを一致判定できない、実行中状態が不整合の場合は「共有未確認」と表示し、ファイル走査や独自履歴移植を行わない。

### UC-10 VS Codeで開始したセッションをWebで確認する

| 項目 | 内容 |
| --- | --- |
| 主アクター | 利用者（VS Code利用後、Deck利用） |
| 目的 | VS Code起点のCodex作業をDeckで閲覧する。 |
| 関連要件 | FR-SS-07、PoC-2 |
| 事前条件 | VS Code拡張とDeckの相互可視性が対象バージョンでPoC合格している。 |
| 成功条件 | Threadを読み取れ、相互操作の保証範囲をUIが明示する。 |

**基本フロー**

1. 利用者はDeckのセッション一覧でsource/workspace/filterを使う。
2. Deckは公式に取得できるThread情報だけを表示する。
3. 利用者は会話、状態、関連ファイル、差分を確認する。
4. Deckでの再開・停止・承認は、PoCで確認された競合規則に従って有効/無効化する。

**例外**

- 同時閲覧/同時承認は、排他が確認されるまで片方をread-only表示にするか、明示的に最新状態を再確認させる。 |

### UC-11 通信断から復帰する

| 項目 | 内容 |
| --- | --- |
| 主アクター | 利用者 |
| 目的 | モバイル通信/Wi-Fi切替やsleepから安全に再接続し、作業の最新状態を得る。 |
| 関連要件 | FR-RT-03〜06、FR-RC-01〜03 |
| 事前条件 | Deckは最終受信event IDと表示位置を端末に保持している。 |
| 成功条件 | 重複を表示せず、欠落イベントまたは最新snapshotで整合状態へ戻る。 |

**基本フロー**

1. ブラウザはsocket切断を検知し、「再接続中」と表示する。
2. CodexのActive workはBridge側で継続する。
3. ブラウザは指数バックオフ+jitterでDeckへ再接続する。
4. 認証後、最後のevent IDからDeck outboxを取得する。
5. gap/順序不整合/保存期限切れならThread、workspace、Git状態をsnapshot再取得する。
6. Deckは未読を更新し、利用者を元の表示位置へ戻す。

**例外**

- App Server自体が切断している場合、Bridgeは再接続/再起動を試行する。実行中Turnを勝手に再送しない。
- オフライン中の下書きは保存できるが、未送信であることを明示する。 |

### UC-12 Windows再起動後に中断作業を確認する

| 項目 | 内容 |
| --- | --- |
| 主アクター | 利用者 |
| 目的 | Windows再起動により中断した作業を誤って再実行せず、文脈を確認して明示再開する。 |
| 関連要件 | FR-RC-04〜05、運用要件 |
| 事前条件 | Deckのタスクスケジューラ起動、DB、ログが復旧している。 |
| 成功条件 | 再起動前にActiveだった作業は「中断」として扱われ、利用者の明示操作なしに再開されない。 |

**基本フロー**

1. Deck Backendが起動し、前回shutdownが正常でなかったことを検出する。
2. Deckは当時Activeだったworkspace/Thread/Turnを中断候補として記録する。
3. 利用者はサーバーステータスまたは通知から候補を開く。
4. Deckは直前イベント、マスク済みログ、Git状態、既存Threadを表示する。
5. 利用者は「再開」「新しい依頼として再送」「確認のみ」を明示選択する。

**禁止事項**

- Deckは前回のprompt、コマンド、承認、破壊的操作を自動再送しない。
- Codex/子プロセスの生存を確認できない場合に、実行継続と表示しない。 |

### UC-13 許可済みworkspaceを追加する

| 項目 | 内容 |
| --- | --- |
| 主アクター | ローカル管理者/利用者 |
| 目的 | 新しい開発フォルダを安全にDeckのworkspaceとして登録する。 |
| 関連要件 | FR-WS-02〜05、SEC-05 |
| 事前条件 | パスが設定済み許可ルート内であり、利用者に登録権限がある。 |
| 成功条件 | path、表示名、Git判定、除外設定、最終利用時刻がDeck補助DBに保存される。 |

**基本フロー**

1. 利用者は「workspaceを追加」を選ぶ。
2. DeckはOSルート全体でなく、許可ルートのフォルダ選択だけを提供する。
3. 利用者は対象フォルダと表示名を確認する。
4. DeckはGit有無、branch、初期ファイル数、deny/mask規則を検査する。
5. Deckは登録し、最近使用/お気に入り情報を初期化する。

**例外**

- 許可ルート外、symlink経由の外部、システム/credential領域は拒否し、理由を表示する。
- 完全自由選択を利用者UIから有効化しない。管理設定の変更は通常のworkspace登録と分ける。 |

## 4. 受入シナリオ

| シナリオ | 合格判定 |
| --- | --- |
| UC-01 | 新規Thread/Turnが開始し、1 workspace 1 Active workの排他が機能する。 |
| UC-02 | タブレットで会話・作業・コード/diffが重ならず、イベントが順序どおり表示される。 |
| UC-03/04 | 回答/承認が一度だけ対象requestに送られ、別クライアント解決時の誤送信を防ぐ。 |
| UC-05/06 | 読み取り専用を保ち、秘密ファイルを出さず、行/diff引用を依頼に使用できる。 |
| UC-07 | parser失敗時もログと終了コードを正しく提示する。 |
| UC-08/11 | ブラウザ閉鎖・通信断後にCodex作業が継続し、再接続で欠落が補完される。 |
| UC-09/10 | PoC合格した表面間だけ、共有可否と制限を正しく表示する。 |
| UC-12 | Windows再起動後に自動再実行せず、中断・ログ・明示再開導線を提供する。 |
| UC-13 | workspace追加が許可ルートを越えず、Git非必須である。 |
